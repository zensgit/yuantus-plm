## 二、结论摘要

1. **平台内核已具备标杆形态，不落后**：元数据驱动类型系统（运行时可管理）、数据驱动生命周期（含条件 DSL）、双轨版本管理（签入签出、分支合并、supersede + 数据库级并发修订护栏）、基线、四类有效性、变型配置器（150% BOM 解算）、状态级权限、合规形电子签名、事务性事件发布（提交后发布、回滚即弃）、四种租户隔离与坐席授权。个别点（并发修订护栏、出箱一致性、租户隔离谱系、图纸查重）超出标杆公开资料所见。
2. **差距不在内核深度，而在四个层面**：
   - **流程谱系宽度**：变更对象只有 ECO 一类（标杆是问题报告/变更请求/变更通告/偏离/让步/返工等约 9 类谱系 + 处置码）；通用工作流引擎停留在原型（无会签权重/委托/超时升级，仅 RPC 可达）；没有跨模块统一待办。
   - **应用广度**：需求工程、正式质量体系（CAPA/FMEA/控制计划/APQP/PPAP）、供应商主数据与 AVL/AML、项目群管理、仿真数据管理、技术出版物、序列化 as-built 数字孪生缺失。
   - **可用性与生态**：通知只有日志桩、无订阅关注与讨论；全文检索不覆盖文档内容；无通用批量导入；表单/网格只读、无菜单导航模型与个人偏好；无 Webhook/SDK；GraphQL 已实现但未挂载；无办公软件集成与图形化批注。
   - **企业级门槛**：无 SSO（OIDC/AD/LDAP）、无配置打包跨环境迁移、服务端脚本未沙箱化、无入站限流、无分布式追踪。
3. **建议节奏**：P0（≤3 个月）10 项快速见效 → P1（3–9 个月）补齐企业门槛 → P2（9–18 个月）铺数字主线。多数工作可复用既有基建（通知/Webhook 需**新增持久投递出箱**、模式借鉴 ERP/ECM 发布出箱；调度器→超时升级；ES 索引器→文档全文；元数据引擎→新对象成本低；授权门控→新模块按版本售卖），整体成本可控。

## 四、分主题差距与优化建议

### 主题 1｜变更谱系与闭环（P0→P1）
**现状**：ECO 很深（`eco_service.py` 3200+ 行），但 ECR 只是接入 DTO（`services/ecr_intake_contract.py`），ECN/偏离/让步/处置码缺失；质量告警与现场破损已有自动生成纠正 ECO 的雏形。
**建议**：
1. 用元数据引擎把 **ECR、ECN 对象化**（新建 ItemType+生命周期+审批钩子；运行时建类型能力已在 `schema_router`，成本主要在业务语义与报表）：ECR 承接问题来源（质量告警/破损/客户反馈），评审后一键派生 ECO（映射函数已有）；ECO 完成自动生成 ECN 并驱动通知与 ERP/ECM 发布（出箱现成）。
2. 增加**偏离/让步**对象，绑定有效性窗口（`effectivity` 已支持日期/批次/序列），审批走 `approvals`。
3. 受影响项引入**处置码**字典（在库/在制/在途/已发运），纳入 ECO 应用诊断与报表。
4. **批量替换向导**：where-used 反查 + 自动生成多父项 `ECOBOMChange`，一次评审批量换料（纯编排，无新模型）。

### 主题 2｜工作流产品化与统一待办（P0）
**现状**：`workflow/` 图引擎（活动/投票/转移条件）是数据驱动的，但投票语义简化（"全部完成取第一个结果"）、无委托/催办/超时升级，且**未挂 REST**（仅 RPC `Workflow.*`）；`approvals/` 是单步审批；ECO 另有独立审批+超时升级任务（`tasks/scheduler_tasks.py`）。
**建议**：
1. 补齐引擎语义：法定人数/权重投票、委托、限期+升级（复用 `SchedulerService` 与 eco_approval_escalation 模式）、催办通知（依赖主题 3）。
2. 挂 REST 路由并纳入路由清单契约测试；流程定义 CRUD（图定义已是 DB 数据，天然可支撑前端可视化设计器）。
3. **统一待办收件箱** `/api/v1/tasks/inbox`：聚合 WorkflowTask、ApprovalRequest、ECO 阶段审批、委外审批交接——标杆实践中"任务收件箱"是用户粘性最高的入口。
4. 长期：ECO 审批逐步收敛到通用引擎，消除双轨。

### 主题 3｜通知、订阅与协作（P0）
**现状**：`notification_service.py` 是 32 行日志桩；无订阅/关注；无讨论。注意：现有通用事件层是**提交后发布到内存 EventBus**（`events/transactional.py:48` → `events/event_bus.py`），无持久化、无重试/死信/投递状态，进程崩溃即丢——只能当触发源，**不能当可靠投递层**（真正的持久出箱只存在于 ERP/ECM 发布模块）。
**建议**：第一刀就新增**持久化 NotificationOutbox/Delivery 表 + 独立投递 worker**（状态机/重试/死信/幂等指纹，模式照搬 `erp_publication`/`ecm_publication` 出箱），内存事件总线仅负责入队触发；投递适配器先邮件 SMTP，后 IM/Webhook；订阅模型（用户×对象/类型×事件类别）+ 摘要合并防打扰；轻量评论/讨论流（可挂任意 item），为评审与变更沟通留痕。

### 主题 4｜检索覆盖文档内容（P0）
**现状**：ES/OpenSearch + SQL 回退很稳（`services/search_service.py`），但只索引物料与 ECO；文档/图纸内容不可搜。实施案例证明"全部文件转 PDF+OCR 再全文索引"是高价值配置。
**建议**：文本抽取管线（PDF 文本层直取；扫描件走已有 OCR 集成契约 `integrations/cad_ml.py`）→ 新增 files 索引（扩展 `search_indexer.py`）→ 搜索 API 增加 `scope=documents`；转换任务复用 `ConversionJob` 队列。

### 主题 5｜分类体系、件号对照与标准件库（P1）
**现状**：无分类树/类属性；标准件只有种子 JSON（`seeder/prod/standard_parts.py`）。实施案例中"400+ 子类分类树 + 按类驱动表单 + 一件多号对照（供应商号/客户号/曾用号）"是落地核心诉求。
**建议**：Classification 树 + 类属性集（`Property` 增加分类作用域）+ 件号别名对象（进检索与查重）；标准件库 API 化并接入分类；表单按分类联动（`ui_layout` 已能承载）。

### 主题 6｜供应商与元器件域（P1）
**现状**：委外闭环完整（订单/回执/处置/审批映射，`subcontracting/`），但 vendor 是自由字符串；无供应商/制造商主数据、AVL/AML、元器件工程。
**建议**：Supplier/Manufacturer/ManufacturerPart 三对象 + Part 上 AVL/AML 关系（"关系即条目"模型天然支持）；委外 `vendor_id` 迁移指向主数据；供应商绩效报表由既有委外分析直接投影；远期做外部器件库联邦查询与停产风险报告（走 `integrations/` 适配器+熔断器模式）。

### 主题 7｜正式质量体系（P1）
**现状**：检验点/检验执行/SPC/质量告警（root_cause 为自由文本）+ 破损→纠正 ECO 闭环；无 CAPA/FMEA/控制计划/APQP/PPAP/8D 结构化对象。
**建议**：三步走——① QualityAlert 升级为 NCR（处置码与变更共用字典）；② CAPA/8D 对象+工作流（复用主题 2 引擎）；③ FMEA 结构（功能/失效模式/影响/措施，RPN 计算）与控制计划（直接挂 `manufacturing` 的工艺路线/工序，模型字段已备）。APQP/PPAP 以模板化交付物清单切入（依赖主题 9）。

### 主题 8｜UI 配置与个性化（P0 起步）
**现状**：Form/GridView/ViewMapping 模型在库、`ui_router` 只读服务；无运行时编辑 API；无菜单/导航树/用户偏好。
**建议**：表单/网格 CRUD 管理端点（照 `schema_router` 的 require_admin_user+ETag 模式）；菜单/导航模型 + 每用户快捷方式与列偏好端点。这些是工作台（消费端）画界面的前置。

### 主题 9｜项目最小闭环（P1）
**现状**：仅 ECO 活动门（DAG 依赖+SLA，`models/parallel_tasks.py`）；无项目/阶段门/交付物。
**建议**：Project/Phase/Gate/Deliverable 四对象，交付物=指向任意 item 的关系；把 ECOActivityGate 的 DAG/SLA 引擎泛化为通用任务门；仪表盘用 `reports` 引擎现成的 Dashboard 模型。先服务 NPD 阶段门评审，不求完整项目管理。

### 主题 10｜配置打包与环境迁移（P1）
**现状**：元数据仅靠 `seed-meta` + 三棵 Alembic 树；租户导入是脚本非产品（`scripts/tenant_import_*.py`）。
**建议**：**元数据包**导出/导入——类型/属性/生命周期/工作流图/权限/表单/报表定义打成版本化 JSON 包，依赖解析复用 `plugin_manager/dependency_resolver`，带 dry-run diff（照 ERP 出箱 dry_run 模式）；CLI+API 双入口。这同时是"客户定制不阻碍升级"承诺的技术基础。

### 主题 11｜开放生态（P0→P1）
**现状**：REST+RPC 齐备；GraphQL 已实现但未挂载（`web/graphql/schema.py:687` 仅测试引用）；无 Webhook；SDK 只有 CAD 桌面插件；无办公集成。
**建议**：P0 决策 GraphQL 挂载（feature flag+entitlement 门控）或删除，避免死代码；Webhook 订阅管理（HMAC 签名+出箱重试）；发布 Python/TS 轻量 SDK（OpenAPI 生成+薄封装）；办公集成先做"模板建文档+属性回填"（编号引擎已有），插件式深度集成放 P2。

### 主题 12｜安全与企业身份（P0→P1）
**现状**：JWT+多组织切换+Ed25519 嵌入令牌是亮点；但无 SSO/AD；Method 脚本 raw `exec` 未沙箱——且有**两条**执行路径：`business_logic/executor.py:44` 与 `services/method_service.py:65`（后者注释自认 production 必须沙箱化）；无入站限流；无属性级权限。
**建议**：P0 安全债拆成两个 PR——**8a Method 脚本沙箱**（受限执行环境或独立进程+资源限额+审计；做**共享 sandbox adapter，两个 exec 入口同时切换**，否则修完仍留旁路）与 **8b 入站限流中间件**（按租户令牌桶），8a 优先；P1 OIDC 对接企业 IdP → AD/LDAP 同步映射到身份库；属性级读写掩码作为权限模型增量（在 `MetaPermissionService` 单点扩展）。

### 主题 13｜数字主线延伸（P2）
**现状**：maintenance（设备/工单/计划）与 MES 消耗对账是 as-built 雏形；无序列化物理件/仿真管理/技术出版物/需求与系统架构。
**建议**：① 序列化 PhysicalPart 从 EBOM 实例化，装机/拆换历史，联动 maintenance 与 MES 形成 as-built/as-maintained 视图；② 仿真数据管理（研究/输入/结果对象挂版本与需求）；③ 结构化技术出版物（内容元素引用 item，引用内容变更→事件总线驱动作者提醒）；④ **需求工程先行试点**（结构化需求元素+复用+与件/验证的追溯矩阵——版本/基线/变更引擎全部现成），再考虑功能/逻辑/物理架构图谱。

### 主题 14｜可观测性与体验补齐（P2）
OpenTelemetry 追踪（中间件栈单点接入）；图形化红线批注（view_state/mesh_stats 已持久化，补 markup 实体+评审联动）；变型解算结果跨修订对比；区域化文件库策略（以 `document_sync` 为基础）。

## 六、实施护栏（沿用本仓库既有纪律）

1. 所有新模块以 **is_entitled 特性键门控**并归入版本/坐席模型；租户配置写路径先管理员校验再授权校验；读投影按"认证→授权→再查询"排序，防存在性泄露。
2. 新路由必须同步**路由清单契约测试**；新测试文件记得加入 CI 显式清单；新环境变量必须在 `Settings` 声明（`extra=ignore` 会静默吞掉未声明项）。
3. Alembic 保持**单一 head**；新对象一律走迁移树。
4. 消费者契约（pact）保持绿色；对外负载变更先跑 dry-run 与契约校验。
5. 脚本沙箱落地前，Method 能力仅限管理员开放，并保持执行审计。

## 七、把标杆的实践痛点变成我们的差异点

从约 60 篇一线实践文章看，标杆平台在真实部署中的高频痛点，恰是我们可以在设计上直接规避的：

| 标杆痛点 | 我们的应对 |
|---|---|
| 界面定制依赖脆弱的 DOM 脚本、随版本失效 | 坚持 `ui_layout`/表单/网格全数据驱动，杜绝客户端 hack |
| 多语言包靠覆盖核心文件、每个版本重做且需备份回滚 | 我们的 `Translation` 模型是数据侧多语言，保持并补齐语言协商 |
| 服务端脚本只能网页编辑、无版本管理、需手动加锁 | Method 走 git 化存储+沙箱执行+审计（主题 12） |
| 部署/打包工具与安装介质解绑、环境迁移易错 | 配置打包做成产品内建能力（主题 10） |
| 许可限制不透明，用户寻求绕过 | entitlement/坐席/版本读数已产品化（license_status），保持透明可视 |
